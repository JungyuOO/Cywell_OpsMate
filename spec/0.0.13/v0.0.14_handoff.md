# v0.0.14 Handoff

## Recommended Implementation Order

1. OpenShift pgvector live database validation
   - Deploy or bind a PostgreSQL instance that has the `vector` extension installed.
   - Start appserver with `CYOPS_PGVECTOR_REQUIRED=true` and `CYOPS_RETRIEVAL_MODE=pgvector`.
   - Confirm startup fails clearly when the extension is missing and succeeds when it is available.

2. Controlled pgvector migration apply
   - Apply `PGVectorEmbeddingMigrationSQL(dimensions)` only after dimensions are fixed.
   - Record rollback guidance for returning to the bytea fallback path in dev/test.

3. RAG smoke over pgvector mode
   - Upload or seed a ready document with chunks and embeddings.
   - Call `/api/chat` with RAG enabled.
   - Verify returned citations come from pgvector SQL ranking.

4. Production metrics export decision
   - Decide whether the operator backend should expose Prometheus format, OpenTelemetry, or structured logs.
   - Keep the v0.0.13 JSON snapshot as an operator diagnostic endpoint unless RBAC review says otherwise.

## Out Of Scope

- ANN index tuning.
- BYOKnowledge indexing.
- Red Hat Lightspeed plugin hiding or deletion.
- High availability for a separate embedding server.

## Acceptance Criteria

- OpenShift pgvector readiness is verified against a live pgvector-enabled database.
- Controlled `VECTOR(n)` migration is applied and documented.
- `/api/chat` RAG path returns pgvector-ranked citations in live mode.
- Retrieval metrics are visible during the live smoke test.
